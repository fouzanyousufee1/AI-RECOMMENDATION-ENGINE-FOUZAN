📋 Overview

This is a full-stack AI-powered movie recommendation application built in four weeks as part of a software engineering assignment. A user types a natural-language mood description and receives the three most contextually relevant movies from a curated 30-film library.

The recommendation engine uses semantic search: every movie description and every mood query is converted into a 1536-dimensional floating-point vector by the OpenAI text-embedding-3-small model. The PostgreSQL pgvector extension computes cosine distance between vectors in pure SQL, making matching fast and precise.

🧠

AI Embeddings

OpenAI converts text to 1536 numbers that encode meaning, not just words.

📐

Vector Search

pgvector's <=> operator finds the closest vectors in milliseconds.

⚙️ How It Works

The recommendation flow from user input to movie results:

User types mood → React frontend → POST /api/recommend → .NET 10 API → mood text → OpenAI API
float[1536] → .NET 10 API → <=> cosine query → PostgreSQL + pgvector → Top 3 rows
JSON response → .NET 10 API → React renders cards

Why embeddings beat keyword search: Compare the mood "something heartwarming for the family" with the movie description for Coco — no words overlap, yet the embedding vectors point in the same direction because both texts share similar meaning. pgvector measures this with cosine distance and the app finds the match automatically.

🔌 API Reference

POST /api/seed

Deletes all existing movies, calls OpenAI once per description (30 total), and inserts all movies with their embedding vectors. Call this once after initial database setup.

Request body: None required.

Success response (200 OK):

{
  "seededCount": 30
}JSON
⚠️ Seeding makes 30 API calls to OpenAI and takes 30–60 seconds. It wipes all existing rows first.
POST /api/recommend

Converts a mood string into a vector, queries pgvector with cosine distance, and returns the three closest matching movies.

Request body:

{
  "mood": "funny and clever"

🗄️ Database Schema

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE movies (
    id          SERIAL PRIMARY KEY,
    title       TEXT NOT NULL,
    genre       TEXT NOT NULL,
    description TEXT NOT NULL,
    embedding   vector(1536),        -- 1536 floats from text-embedding-3-small
    created_at  TIMESTAMP DEFAULT NOW()
);SQL

The embedding vector(1536) column stores one float per embedding dimension. pgvector adds the <=> cosine distance operator directly in SQL, so no external library is needed at query time.

💻 Key Code Explained

1 · Generating an embedding (MoviesController.cs)

The controller uses the OpenAI SDK to convert any text string into a Vector ready for PostgreSQL:

// Constructor: build EmbeddingClient from injected OpenAIClient
_embeddingClient = openAIClient.GetEmbeddingClient("text-embedding-3-small");

// Helper method: text → Vector
private async Task<Vector> GenerateVectorAsync(string text, CancellationToken ct)
{
    var result = await _embeddingClient.GenerateEmbeddingAsync(text, cancellationToken: ct);
    float[] floats = result.Value.ToFloats().ToArray();  // 1536 floats
    return new Vector(floats);                          // Pgvector.Vector
}C#

2 · pgvector cosine distance query (MoviesController.cs)

The recommend endpoint executes raw SQL with the <=> operator:

SELECT title, genre, description,
       1 - (embedding <=> @queryVec::vector) AS score
FROM movies
ORDER BY embedding <=> @queryVec::vector
LIMIT 3;SQL
<=> returns cosine distance (0 = same direction, 2 = opposite). Subtracting from 1 converts it to cosine similarity so higher score = better match.

3 · Multi-port API fallback (App.jsx)

The frontend tries the backend on multiple ports before giving up:

const API_BASES = [
  import.meta.env.VITE_API_BASE_URL,   // optional env var override
  "http://localhost:5053",              // primary backend port
  "http://localhost:5000"               // fallback
].filter(Boolean);

for (const base of API_BASES) {
  try {
    const res = await fetch(`${base}/api/recommend`, { /* ... */ });
    if (res.ok) { /* use result and break */ break; }
  } catch { /* try next port */ }
}JSX

4 · CORS policy (Program.cs)

The backend allows any localhost port — no config change needed when the frontend port changes:

policy.SetIsOriginAllowed(origin => {
    if (!Uri.TryCreate(origin, UriKind.Absolute, out var uri)) return false;
    return uri.Host.Equals("localhost", StringComparison.OrdinalIgnoreCase);
})C#

5 · Movie model (MovieContext.cs)

The EF Core entity maps directly to the PostgreSQL movies table, including the pgvector column:

public class Movie
{
    public int     Id          { get; set; }
    public string  Title       { get; set; } = "";
    public string  Genre       { get; set; } = "";
    public string  Description { get; set; } = "";
    public Vector? Embedding   { get; set; }  // Pgvector.Vector — maps to vector(1536)
    public DateTime CreatedAt  { get; set; }
}C#

🔒 Security Notes

🔑

No secrets in git

appsettings.Development.json is gitignored. Only the .example template is committed.

🌐

CORS localhost-only

The CORS policy only allows localhost origins. Do not use AllowAnyOrigin in production.

🛡️

Parameterised SQL

All database queries use named parameters via Npgsql. No string concatenation in SQL — no SQL injection risk.

✅

Input validation

The recommend endpoint returns 400 if mood is null or whitespace before any DB or API call is made.
